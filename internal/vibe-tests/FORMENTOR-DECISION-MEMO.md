# Decision Memo: Which form framework should be Astryx's default?

**Question.** Astryx components are excellent, but Astryx has no opinion on form
state/validation. Should it adopt **Formentor** (a schema-aware layer built for
Astryx) as the blessed default, or bless an existing library — **Formisch**,
**TanStack Form**, or **React Hook Form** — with a thin Astryx adapter?

**Method.** A 4-way "vibe test" holding the design system constant (every target
renders the *same* Astryx inputs) and varying only the form-framework layer. 15
realistic, naive CORE prompts × 4 frameworks = 60 forms, each generated by a
**blank-slate agent** that learns the API cold from the package README (invariant
#6). Scoring is deterministic and **identity-blind** (invariant #7): a tsc
compile check plus five form-specific dimensions, with equal-quality reference
solutions calibrated to score within a tight band across all four frameworks
(they tie at 100 — zero rubric bias). Full spec: `specs/formentor-level-b.md`.

Each non-Formentor framework got an **equal-effort Astryx adapter** (Option B
parity): the same five bridge components binding to the same core inputs. No
framework was pre-integrated more than another; none was hobbled.

## Results (CORE tier, headline)

| Framework | Clean tsc compile | Form-quality score | Median LOC | Avg doc lookups |
|-----------|------------------:|-------------------:|-----------:|----------------:|
| **Formentor** | 87% (13/15) | 98 | **49** | **4.5** |
| **Formisch**  | 93% (14/15) | 100 | 56 | 5.9 |
| **TanStack**  | 93% (14/15) | 100 | 75 | 6.2 |
| **React Hook Form** | 87% (13/15) | 100 | 57 | 5.1 |

All four cluster within one prompt on compile rate and are near-tied on
form-quality. The separating signals are **conciseness** and **cognitive load**,
not correctness.

### What the numbers mean (after removing two confounds)

The first raw run showed TanStack/RHF at 60% compile — but every one of those
failures was a **Zod v3-vs-v4 API mismatch** (the harness resolved a hoisted Zod
v4 while the adapters install v3; agents wrote v3 syntax from training data), not
a framework difference. Pointing the typecheck at the v3 copy the adapters
actually resolve, and widening the RHF adapter's `control` type to accept
transform resolvers, lifted both to a fair 87–93%. This is exactly the kind of
asymmetry the fairness invariants exist to catch, and it is documented in the
harness README.

The residual failures are **genuine**:
- **Formentor (2):** one agent bug (`{}` passed to a string validator); one used
  `defaultValue` (the README documents `initialValue`).
- **RHF (2):** `z.literal(true)` for a required checkbox makes RHF infer the
  field as literally `true`, which then fights `defaultValues: false` — a real
  RHF+Zod sharp edge a naive author hits.
- **TanStack (1) / Formisch (1):** optional-field inference friction.

## Qualitative findings (from reading all 60 solutions)

**Formentor is the terse, low-ceremony option.** Its schema-first model means a
login form is 34 lines and `form.render()` handles layout, validation wiring, and
the submit button. Agents needed the fewest doc lookups (4.5) and almost never
hand-rolled state. The schema *is* the form. For the common case (scalars,
required/optional, format + length validation, cross-field, async submit) it was
consistently the shortest correct answer, and its native `mode: 'view' | 'edit'`
made the view/edit prompt trivial where others hand-rolled an `isEditing` toggle.

**TanStack is the most verbose.** `createFormHook` boilerplate + per-field
`form.AppField` render-props pushed it to a 75-line median (same login: 59 lines).
It is powerful and its `form.Subscribe`/`canSubmit` gating is elegant, but it asks
the most of a naive author.

**Formisch and RHF sit in between.** Both are schema-first (Valibot / Zod), both
idiomatic, both familiar to the models. RHF's `Controller` + `zodResolver` is the
most-recognized pattern; Formisch's Valibot-native `v.forward`/`v.check` handled
cross-field cleanly.

**One real Formentor gap surfaced:** its `form.state.isValid` only reflects
touched/submitted fields, so the "disable submit until valid" prompt forced a
hand-rolled completeness gate — where RHF's `formState.isValid` and TanStack's
`state.canSubmit` work out of the box. Worth fixing before Formentor is the
default.

### Scaffold improvements this test forced (blank-slate agents found real gaps)

- `ObjectSchemaImpl.getValuePath()` — documented in the README's cross-field
  example but didn't exist until an agent followed the docs and hit a type error.
- Field-level `render({validate})` callbacks left `pass`/`fail` implicitly `any`;
  now typed.
These are exactly the kind of paper-cuts a concept scaffold hides until real,
uncoached users exercise it.

## Recommendation

**Adopt Formentor as the default, but gate it on two fixes.** On a constant
Astryx design system, Formentor produced the most concise, lowest-cognitive-load
forms for the common case and matched the field on correctness and form quality —
while being the only option that unifies schema, validation, layout, and
view/edit into one artifact. That is the differentiated value; the others are
excellent general-purpose libraries but leave the schema↔form↔UI seam to the
author.

Gate the default on:
1. **Fix `isValid` semantics** so "disable until valid" works without a
   hand-rolled gate (the one real ergonomic miss).
2. **Grow CORE coverage.** This test only exercised Level-B scalars. Formentor
   must earn the STRETCH tier — field arrays, nested objects, conditional fields,
   wizards, typeahead, file upload — before it's the default for *all* forms.
   Until then, recommend Formentor for standard forms and keep the RHF or Formisch
   adapter available as the escape hatch for advanced cases.

**If Formentor cannot absorb the STRETCH scope soon,** the strongest
already-complete alternative is **Formisch**: schema-first like Formentor, tied
for the best compile rate, terser than TanStack/RHF, and multi-framework.

## Caveats

- CORE only — 15 prompts, Level-B scalar forms. STRETCH (field arrays, nested,
  wizards, typeahead, upload) was **not** run; Formentor's scaffold does not yet
  support it. Headline numbers do not speak to advanced forms.
- One generation per (prompt × framework); no repeat sampling for variance.
- Deterministic + tsc scoring only. The vision/design judge (rendered-output
  fidelity) was deferred and is not in these numbers.
- Formentor is a Level-B *concept scaffold*, not the internal WWW Formentor; the
  server-side schema-derivation half (Thrift/Ent/Hack → codegen) is out of scope.
